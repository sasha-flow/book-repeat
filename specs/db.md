# Database

## Overview

This document is the canonical reference for Book Repeat database structure, storage-backed data ownership, and database-side business rules.

If another document needs to mention persistence, it should describe only behavior and link back to this file for schema details.

The current persisted model is implemented through the SQL migrations in `supabase/migrations` and backed by Supabase Postgres, Supabase Auth, and Supabase Storage.

The local Supabase configuration currently targets PostgreSQL major version `17`, and local development should stay on the same major version as the remote project so `supabase db reset`, `supabase db push`, and CI dry-runs behave consistently.

## Source of truth

- application schema changes are migration-driven through `supabase/migrations`
- the current source-reader SQLite reference is `specs/sqlite-db-structue.sql`
- this file documents the effective target schema after all tracked migrations are applied

## Target entities

### books

Purpose:

- stores one canonical logical book per user after import reconciliation
- acts as the stable application identity that bookmarks reference

Fields:

- `id`: UUID primary key generated by Postgres
- `user_id`: owning Supabase auth user, foreign key to `auth.users(id)`, cascade delete
- `title`: imported book title used in browsing and reading UI
- `authors`: flattened author list stored as text, defaults to empty string
- `created_at`: row creation timestamp
- `updated_at`: row update timestamp maintained by trigger

Business meaning:

- a row in `books` is not one source hash and not one upload artifact
- it represents the app's current best canonical identity for one logical book owned by one user
- multiple source hashes may point to the same canonical book through `book_source_hashes`

Indexes and constraints:

- primary key on `id`
- index `books_user_id_idx` on `user_id`

### book_source_hashes

Purpose:

- stores every imported source `BookHash.hash` value that has been linked to a canonical book
- provides the reconciliation layer that allows multiple devices and multiple exports to collapse into one canonical book

Fields:

- `id`: UUID primary key generated by Postgres
- `user_id`: owning Supabase auth user, foreign key to `auth.users(id)`, cascade delete
- `book_id`: canonical book reference, foreign key to `books(id)`, cascade delete
- `source_hash`: source-reader hash imported from the SQLite `BookHash` table
- `created_at`: alias creation timestamp

Business meaning:

- uniqueness on `(user_id, source_hash)` guarantees that one source hash belongs to at most one canonical book for a given user
- many rows may point at the same `book_id`
- this table is the core of cross-device deduplication and canonical-book auto-merge behavior

Indexes and constraints:

- primary key on `id`
- unique constraint on `(user_id, source_hash)`
- index `book_source_hashes_user_id_idx` on `user_id`
- index `book_source_hashes_book_id_idx` on `book_id`

### bookmarks

Purpose:

- stores imported bookmarks and highlights for reading inside the app
- links each bookmark to a canonical book

Fields:

- `id`: UUID primary key generated by Postgres
- `user_id`: owning Supabase auth user, foreign key to `auth.users(id)`, cascade delete
- `source_uid`: stable bookmark identity from the source SQLite database
- `book_id`: canonical book reference, foreign key to `books(id)`, cascade delete
- `bookmark_text`: bookmark body text shown to the reader
- `paragraph`: source paragraph position used for ordering
- `word`: source word position used for ordering
- `bookmark_type`: enum `default | header | hidden`
- `source_style_id`: copied source style id for traceability and reclassification
- `source_visible`: copied source visibility flag for traceability and reclassification
- `source_creation_time`: copied source creation timestamp
- `source_modification_time`: copied source modification timestamp
- `created_at`: row creation timestamp
- `updated_at`: row update timestamp maintained by trigger

Business meaning:

- uniqueness on `(user_id, source_uid)` makes repeated imports idempotent for the same source bookmark
- `book_id` always points to the current canonical book after any automatic merge

Indexes and constraints:

- primary key on `id`
- unique constraint on `(user_id, source_uid)`
- index `bookmarks_book_id_idx` on `(book_id, paragraph, word)`
- index `bookmarks_user_id_idx` on `user_id`

### import_runs

Purpose:

- stores a lightweight audit trail of completed import attempts
- supports operational debugging and future import history UI

Fields:

- `id`: UUID primary key generated by Postgres
- `user_id`: owning Supabase auth user, foreign key to `auth.users(id)`, cascade delete
- `file_name`: uploaded SQLite filename
- `books_count`: number of canonical book groups written during the import
- `bookmarks_count`: number of bookmark rows written during the import
- `delete_error`: storage cleanup error message if uploaded file deletion failed
- `created_at`: import completion timestamp

Indexes and constraints:

- primary key on `id`
- index `import_runs_user_id_idx` on `(user_id, created_at desc)`

## Enums and functions

### bookmark_type

Values:

- `default`
- `header`
- `hidden`

Business meaning:

- `default` is normal reading content
- `header` is visible but excluded from the strict reading filter
- `hidden` is excluded from the visible and reading filters unless the user explicitly shows all content

### set_updated_at()

Purpose:

- central trigger function that refreshes `updated_at` on mutable tables

Used by:

- `books_updated_at`
- `bookmarks_updated_at`

### merge_user_books(target_user_id, winner_book_id, loser_book_ids)

Purpose:

- database-side merge primitive used by the import pipeline when one incoming hash set overlaps multiple existing canonical books

Behavior:

- validates that the winner and losers belong to the same target user
- moves non-conflicting hash aliases from loser books to the winner book
- removes duplicate loser aliases that would conflict after the move
- rewrites bookmarks from loser books to the winner book
- deletes loser book rows
- returns JSON counters for moved aliases, deleted aliases, moved bookmarks, and deleted books

Reason for living in the database:

- keeps multi-table merge behavior close to the data
- reduces race-condition risk versus piecemeal client-side merge steps
- gives the import route a deterministic primitive for canonical-book reconciliation

## Relationship model

- one `auth.users` row owns many `books`
- one `auth.users` row owns many `book_source_hashes`
- one `auth.users` row owns many `bookmarks`
- one `auth.users` row owns many `import_runs`
- one `books` row owns many `book_source_hashes`
- one `books` row owns many `bookmarks`

## Import mapping from source SQLite

The source SQLite schema reference is `specs/sqlite-db-structue.sql`.

Current import interpretation:

- each source `Books.book_id` is parsed as one source-book candidate
- every `BookHash.hash` row for that source `book_id` is retained
- `Books.title` maps to `books.title`
- `Authors.name` values are flattened in `BookAuthor.author_index` order into `books.authors`
- `Bookmarks.uid` maps to `bookmarks.source_uid`
- `Bookmarks.book_id` first links a bookmark to a source book, then the import planner resolves that source book to a canonical `books.id`
- `Bookmarks.bookmark_text`, `paragraph`, and `word` map directly into `bookmarks`
- source `visible` and `style_id` values are normalized into `bookmark_type`, while original values are also persisted for traceability

## Canonical-book reconciliation rules

- canonical books are matched by any overlapping `source_hash` already present in `book_source_hashes`
- if no hashes overlap, a new canonical `books` row is created
- if exactly one canonical book overlaps, that book is reused
- if multiple canonical books overlap, the import pipeline merges them into one deterministic winner and rewires aliases and bookmarks to that winner
- after reconciliation, every imported hash is upserted into `book_source_hashes`
- after reconciliation, every imported bookmark is written against the resolved canonical `book_id`

## Security model

Row-level security is enabled on:

- `books`
- `book_source_hashes`
- `bookmarks`
- `import_runs`

Policy coverage:

- `books`: owner-scoped `select`, `insert`, `update`, and `delete`
- `book_source_hashes`: owner-scoped `select`, `insert`, `update`, and `delete`
- `bookmarks`: owner-scoped `select`, `insert`, `update`, and `delete`
- `import_runs`: owner-scoped `select` and `insert`

Storage security:

- uploaded import files live in the private `imports` bucket
- storage object policies restrict `insert`, `select`, and `delete` to the authenticated user's own folder prefix inside `imports`

Operational security note:

- the browser app is expected to use these owner-scoped RLS and storage policies directly for normal reads and bookmark-type updates
- the import route uses a validated service-role client for privileged writes, but the user-scoped policies still define the steady-state access model for persisted data

## Operational notes

- uploaded SQLite files are temporary processing artifacts and should be deleted after import processing
- the application reads through RLS-scoped browser clients for normal user actions
- the import route uses a service-role client only after validating the user token
- production schema rollout is migration-driven, not manually edited in the Supabase dashboard
